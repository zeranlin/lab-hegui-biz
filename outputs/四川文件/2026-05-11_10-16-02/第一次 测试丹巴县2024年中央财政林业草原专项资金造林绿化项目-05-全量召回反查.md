## 5. 是否命中物业管理专项动作协议

**否**

*理由*：本项目标的为“造林绿化项目”，主要采购内容为苗木、肥料、标识牌及少量人工管护，属于农林物资采购，不属于“物业管理服务”（保洁、安保、绿化养护、机电运维等）。虽然包含“人工管护”，但其核心是造林后的初期管护，而非物业小区的绿化养护。因此，不触发 [[wiki/70-审查协议/物业管理审查动作协议]] 及 [[wiki/20-知识点/物业管理动作化审查包]]。

## 6. 路由结论

本项目为**四川省丹巴县**的**公开招标**项目，标的为**农林物资（苗木/肥料/标识牌）及少量管护服务**，**专门面向中小企业**。

**路由策略**：
1.  **基础层**：调用全国通用政府采购规则（87号令、中小企业政策）及四川省/丹巴县地方采购规定。
2.  **核心层**：调用**货物类**采购审查规则，重点审查技术参数指向性、中小企业声明函一致性、所属行业判定。
3.  **场景层**：调用**林业/农林物资**场景规则，重点审查验收标准（成活率）、管护服务边界、专项资金合规性。
4.  **排除层**：明确排除物业管理、信息化、建筑工程等品目的专项审查包。

# wiki/00-入口/外部执行主体招标文件审查指引.md

# 外部执行主体招标文件审查指引

类型:: 指引
状态:: 有效
主题:: 外部执行主体; 招标文件审查; 政府采购风险识别
适用对象:: 可读取本 LLM Wiki 的 AI Agent、自动化执行器、脚本系统、人工审查助理
任务目标:: 输入一份政府采购招标文件，输出 AI 审查记录和运行记录
关联流水线:: [[wiki/70-审查协议/政府采购招标文件业务审查流水线]]
关联执行规范:: [[wiki/70-审查协议/知识驱动审查执行规范]]
关联质量门:: [[wiki/70-审查协议/质量门规则]]

## 1. 你要完成什么

你收到的是一份政府采购招标文件。你的任务不是总结文件，而是识别合规风险点。

你必须读取本 LLM Wiki 的知识，按照本指引完成：

```text
招标文件
  -> 文件画像
  -> 知识路由
  -> 动作清单
  -> 逐章逐动作审查
  -> 风险点识别
  -> 风险原子化
  -> 质量门反查
  -> AI 审查记录
  -> 运行记录
```

最终输出应类似：

```text
物业管理-[项目编号]项目名称 - AI 审查记录
```

并同时输出一份运行记录，证明你确实按知识系统完成了审查。

## 2. 必读知识顺序

外部执行主体初始化审查时，按以下顺序读取：

| 顺序 | 必读内容 | 目的 |

| 5 | [[wiki/70-审查协议/政府采购招标文件审查协议]] | 获取通用审查协议和运行记录要求 |
| 6 | [[wiki/20-知识点/政府采购招标文件画像]] | 学会先建立文件画像 |
| 7 | [[wiki/20-知识点/知识分层与路由规则]] | 根据画像选择知识 |

不要只读风险审查点总览后直接生成结论。必须先画像、路由、逐章审查。

## 3. 审查步骤

### 3.1 读取原文

读取招标文件全文或抽取文本。

必须保留：

- 页码、行号、条款号或章节位置；
- 表格结构；
- 金额、编号、日期；
- 空白字段；
- 合同附件和格式文件。

如果无法抽取原文，应停止审查并输出失败原因。

### 3.2 建立文件画像

先判断文件是什么，再判断风险。

画像至少包括：

- 项目编号；
- 项目名称；
- 采购人；
- 代理机构；
- 采购方式；
- 评审方法；
- 预算金额和最高限价；
- 采购品目；
- 采购标的；
- 标的所属行业；
- 是否专门面向中小企业；
- 是否接受联合体；
- 是否允许分包；
- 是否允许进口产品；
- 场景标签；
- 文件章节完整性；
- 初步风险入口。

画像示例：

```text
地域=深圳市
采购方式=公开招标
标的属性=服务
品类=物业管理
场景=学校物业
评审方法=综合评分法
```

### 3.3 路由知识

根据画像读取对应知识。

路由顺序：

```text
全国通用规则
  -> 地方规则
  -> 采购方式规则
  -> 标的属性规则
  -> 品类规则
  -> 场景规则
  -> 章节规则
```

如果画像命中物业管理，必须读取：

- [[wiki/15-行业基础/物业管理服务采购背景]]
- [[wiki/20-知识点/物业管理动作化审查包]]
- [[wiki/70-审查协议/物业管理审查动作协议]]

### 3.4 生成动作清单

动作清单不是自由发挥，应来自动作协议。

每个动作至少包括：

```text
动作ID
动作名称
必读章节
触发信号
必须检查
执行状态：命中 / 未命中 / 待确认
关联审查点
输出要求
```

物业管理文件必须生成并执行 `PROP-A01` 至 `PROP-A23`。

### 3.5 逐章逐动作审查

按章节读取原文，并执行对应动作：

| 章节 | 必查重点 |
| --- | --- |
| 招标公告 | 资格条件、采购方式、预算、联合体、中小企业政策 |
| 采购计划和预算 | 采购计划编号、预算金额、最高限价、标的明细、报价口径 |
| 政府采购政策 | 中小企业、进口产品、节能环保、声明函 |
| 资格和符合性 | 资格条件、分公司、关联主体、实质性条款、投标无效 |
| 用户需求书 | 服务范围、人员配置、费用边界、响应时限、验收考核 |
| 评分办法 | 响应度、方案、人员、认证、资质、业绩、荣誉、软著 |
| 证明材料 | 社保、查询截图、原件备查、合同关键页、证书材料 |
| 合同条款 | 服务范围、付款、验收、违约、模板错配、交接责任 |

每章都要有审查结论。没有发现风险，也要说明已审查。

如果文件画像命中物业管理，必须执行以下动作：

| PROP-A11 | 服务团队证书组合 |
| PROP-A12 | 服务团队社保时点 |
| PROP-A13 | 证明材料和原件备查 |
| PROP-A14 | 体系认证范围 |
| PROP-A15 | 供应商资质证书 |

## 5. 风险点判断

每个风险点必须同时具备：

- 原文证据或明确缺失事实；
- 风险判断理由；
- 关联动作ID；
- 关联风险审查点；
- 审查依据；
- 修改建议。

不能只凭模型印象判断风险。

依据不确定时，写：

```text
需进一步确认
```

## 6. 风险原子化

不要把多个问题写成一条大风险。

以下必须拆分：

- 人员证书组合和社保时点；
- 人员证书组合和证明材料；
- 用户需求人员限制和评分人员证书；
- 体系认证和供应商资质；
- 业绩范围和业绩证明材料；
- 合同模板错配和付款验收违约空白；
- 承诺函评分和合同条款不清。

按 [[wiki/70-审查协议/风险原子化规则]] 执行。

## 7. 风险点输出格式

每条风险使用以下字段：

```text
### 编号. 风险标题

- 结论类型：
- 风险等级：
- 原文位置：
- 原文内容：
- 问题说明：
- 关联动作ID：
- 关联审查点：
- 审查依据：
- 修改建议：
- 是否需要人工复核：
```

字段要求：

- `原文内容` 只能放招标文件原文；
- `问题说明` 写风险成立理由；
- `关联动作ID` 对应动作协议；
- `关联审查点` 对应 `wiki/25-风险审查点/`；
- `修改建议` 必须可执行。

## 8. AI 审查记录输出

AI 审查记录至少包含：

```text

状态:: AI 审查版
项目编号::
项目名称::
采购人::
代理机构::
采购方式::
评标方法::
预算金额::
采购品目::
采购标的所属行业::
审查日期::
审查时间::
审查人::
审查状态::
外部标注使用:: 否
风险点数量::
运行记录::
```

正文至少包含：

- 审查摘要；
- 文件画像；
- 逐章审查记录；
- 风险点清单；
- 已审查未列风险；
- 待补证或待确认；
- 文件位置。

参考样板：

- [[wiki/40-审查工作台/审查记录/物业管理/[BACG2025000216-A]深圳市宝安区西湾小学蘅芳校区物业管理服务-审查记录]]

## 9. 运行记录输出

运行记录用于证明你不是直接总结，而是按知识执行。

运行记录至少包含：

- 基本信息；
- 执行边界；
- 调用知识清单；
- 文件画像；
- 动作清单和执行状态；
- 原文读取记录；
- 风险原子化检查；
- 质量门结果；
- 未命中动作说明；
- 待人工复核事项；
- 是否使用外部标注；
- 是否修改知识库；
- 底层模型调用次数无法取得时的说明。

参考样板：

- [[wiki/40-审查工作台/运行记录/物业管理/[BACG2025000216-A]深圳市宝安区西湾小学蘅芳校区物业管理服务-AI调度运行记录]]
- [[wiki/40-审查工作台/运行记录/物业管理/[BACG2025000216-A]深圳市宝安区西湾小学蘅芳校区物业管理服务-业务审查流水线执行审计证明]]

## 10. 质量门

输出前必须执行质量门。

最低检查：

- 文件画像是否完整；
- 知识路由是否完整；
- 动作清单是否全部有状态；
- 是否逐章审查；
- 每条风险是否有原文内容；
- `原文内容` 是否只保留原文；
- 每条风险是否有关联动作ID；
- 每条风险是否有关联审查点；
- 人员评分是否检查证书、社保、证明材料；
- 供应商能力是否检查认证、资质、业绩、荣誉、软著；
- 用户需求是否检查人员限制、服务边界、响应时限；
- 合同是否检查模板错配、付款、验收、违约；
- 风险数量明显偏低时是否反查漏项。

质量门不通过时，不得输出最终审查记录。

## 11. 禁止事项

外部执行主体不得：

- 使用外部 V2、标准答案、人工批注反推风险；
- 只读摘要，不读原文；
- 跳过文件画像；
- 跳过知识路由；
- 只按关键词检索风险；
- 把多个应拆分风险合并成一条；
- 把问题说明写进 `原文内容`；
- 生成无原文证据的风险；
- 跳过质量门；
- 自动改写稳定知识库；
- 生成第二套校准版审查记录冒充 AI 审查结果。

## 12. 初始化成功标准

外部执行主体只有满足以下条件，才算成功初始化：

- 能说明本任务输入是一份政府采购招标文件；
- 能读取本指引列出的必读知识；
- 能建立文件画像；
- 能根据画像路由品类动作协议；
- 能生成动作清单；
- 能逐章逐动作读取原文；
- 能输出风险点并关联动作ID和审查点；
- 能执行风险原子化；
- 能执行质量门；
- 能生成 AI 审查记录；
- 能生成运行记录。

如果只能输出一段总结或少量风险，但没有画像、动作清单、质量门和运行记录，视为未成功初始化。

- 审查记录和运行记录模板；
- 审后复盘和知识回写方法。

外部执行主体的职责是读取这些知识，并把知识转成自己的执行流程。外部执行主体不应要求本知识库为某个具体工具生成项目化开发任务。

## 2. 知识读取顺序

外部执行主体审查新文件时，应按以下顺序读取知识：

```text
AGENTS.md
  -> wiki/index.md
  -> wiki/00-入口/动作型知识系统建设纲领
  -> wiki/20-知识点/政府采购招标文件画像
  -> wiki/20-知识点/知识分层与路由规则
  -> wiki/20-知识点/政府采购逐章审查矩阵
  -> wiki/70-审查协议/政府采购招标文件审查协议
  -> 品类动作协议
  -> 风险原子化规则
  -> 质量门规则
  -> 审查记录和运行记录模板
```

如果文件画像命中物业管理，应额外读取：

- [[wiki/15-行业基础/物业管理服务采购背景]]
- [[wiki/20-知识点/物业管理动作化审查包]]
- [[wiki/70-审查协议/物业管理审查动作协议]]
- 与画像命中的场景对应的风险审查点和知识点。

## 3. 执行链路

知识驱动审查不是“检索几条知识后让模型自由总结”，而是以下链路：

```text
读取原文
  -> 建立文件画像
  -> 根据画像路由知识
  -> 生成本次审查动作清单
  -> 按章节和动作读取原文
  -> 对每个动作形成命中/未命中/待确认
  -> 将命中项拆成原子风险
  -> 补足原文证据和审查依据
  -> 执行质量门反查
  -> 生成审查记录
  -> 生成运行记录
  -> 将可复用漏报误报进入复盘和知识更新流程
```

## 4. 文件画像要求

文件画像是知识路由的前置动作。外部执行主体不得跳过画像直接输出风险。

画像至少包括：

| 维度 | 必填内容 |
| --- | --- |
| 基本信息 | 项目编号、项目名称、采购人、代理机构 |
| 程序属性 | 采购方式、评审方法、是否评定分离、提交投标截止时间 |
| 标的属性 | 采购品目、采购标的、数量、单位、标的所属行业 |
| 金额信息 | 预算金额、最高限价、分项预算、采购计划编号 |
| 政策适用 | 中小企业、节能环保、进口产品、绿色采购、核心产品 |
| 组织规则 | 联合体、分包、分公司、关联主体、资格条件 |
| 文件结构 | 公告、须知、资格、需求、评分、合同、格式是否完整 |
| 场景标签 | 学校物业、医院物业、法院机关物业、信息化设备等 |
| 初步入口 | 缺失项、异常人员、异常证书、异常评分、合同空白 |

画像输出必须反向说明：本次为什么调用某些知识点、风险审查点和动作协议。

## 5. 知识路由要求

知识路由应按层级执行：

```text
全国通用规则
  -> 地方规则
  -> 采购方式规则
  -> 标的属性规则
  -> 品类规则
  -> 场景规则
  -> 章节规则
  -> 试运行规则
```

外部执行主体应记录本次调用的知识清单。记录内容至少包括：

- 调用的法规依据；
- 调用的行业基础；
- 调用的知识点；
- 调用的风险审查点；
- 调用的审查协议；
- 调用的模板；
- 未调用但可能相关的知识及原因。

## 6. 动作清单生成

动作清单来自审查协议和品类动作协议，不由外部执行主体凭经验临时编造。

动作清单至少包含：

| 字段 | 要求 |
| --- | --- |
| 动作ID | 来自动作协议，例如 `PROP-A09` |
| 动作名称 | 与动作协议一致 |
| 必读章节 | 明确原文范围 |
| 触发信号 | 关键词、缺失字段、章节结构或画像标签 |
| 必须检查 | 成立条件和排除条件 |
| 执行状态 | 命中、未命中、待确认 |
| 命中风险 | 对应风险标题或候选 |
| 未命中说明 | 已读范围和不成立原因 |
| 关联审查点 | 对应 `wiki/25-风险审查点/` |
| 人工复核 | 是否需要人工确认 |

物业管理项目应从 [[wiki/70-审查协议/物业管理审查动作协议]] 生成 `PROP-A01` 至 `PROP-A23` 的动作清单。

## 7. 原文读取要求

外部执行主体必须按章节和动作读取原文，不得只读摘要或只按关键词命中。

最低读取范围：

- 招标公告；
- 采购计划、预算和服务需求明细；
- 政府采购政策和声明函；
- 资格条件、联合体、分包、分公司；
- 资格性审查和符合性审查；
- 用户需求书；
- 评分办法；
- 证明材料和投标文件格式；
- 合同条款；
- 付款、验收、违约和考核条款。

读取结果应能支持原文定位，优先保留行号、页码、条款号或表格位置。

## 8. 风险判断要求

风险成立不得只靠模型感觉。每条风险至少满足：

- 有原文证据或明确缺失事实；
- 有关联风险审查点；
- 有风险成立理由；
- 有排除条件判断；
- 有法规、知识点或审查协议依据；
- 有可执行修改建议；
- 能被人工单独确认、驳回、降级、合并或拆分。

法律依据不确定时，标记为 `需进一步确认`，不得强行定性。

## 9. 风险原子化要求

外部执行主体必须按 [[wiki/70-审查协议/风险原子化规则]] 拆分风险。

特别注意以下不得合并：

- 人员证书组合和人员社保时点；
- 人员证书组合和证明材料负担；
- 需求端人员限制和评分端人员证书限制；
- 体系认证范围和供应商资质等级证书；
- 同类业绩范围和业绩证明材料；
- 合同模板错配和付款、验收、违约空白；
- 响应度评分、承诺函评分和方案类主观分档。

如果外部执行主体选择合并风险，必须说明合并理由，并保证合并后仍能分别整改。

## 10. 输出字段要求

每条风险至少包含：

```text
风险标题
结论类型
风险等级
原文位置
原文内容
问题说明
关联动作ID
关联审查点
审查依据
修改建议
是否需要人工复核
```

字段边界：

- `原文内容` 只放招标文件原文；
- `问题说明` 放判断理由；
- `关联动作ID` 放动作协议中的动作编号；
- `关联审查点` 放稳定风险审查点；
- `修改建议` 放可执行修订动作。

## 11. 质量门要求

外部执行主体输出前必须执行 [[wiki/70-审查协议/质量门规则]]。

最低质量门：

- 文件画像是否完整；
- 知识路由是否完整；
- 动作清单是否全部有状态；
- 每条风险是否有原文证据；
- 每条风险是否有关联动作ID和审查点；
- 是否存在应拆未拆的风险；
- 人员评分是否反查证书、社保、证明材料；
- 供应商能力评分是否反查认证、资质、业绩、荣誉、软著；
- 用户需求是否反查人员限制、校外边界、响应时限；
- 合同条款是否反查模板错配、付款、验收、违约；
- 风险数量异常偏低时是否触发漏项反查。

质量门结果应写入运行记录。

## 12. 运行记录要求

运行记录用于证明审查过程可追溯、可复现。

运行记录至少包含：

- 审查日期和具体时间；
- 输入文件；
- 使用的知识库路径；
- 调用知识清单；
- 文件画像；
- 动作清单及执行状态；
- 逐章读取范围；
- 风险原子化检查；
- 质量门结果；
- 未命中动作说明；
- 待人工复核事项；
- 是否使用外部参考；
- 是否修改知识库。

## 13. 权限边界

外部执行主体读取本知识库时，应遵守：

- `raw/` 原始文件只读；
- 稳定法规、知识点、风险审查点、风险库、审查协议只读；
- 审查产物写入项目审查记录或外部系统自己的输出目录；
- 审后发现的可复用漏报误报，应进入人工复盘和知识更新申请流程；
- 不得把外部标准答案、人工批注或 V2 检查结果混入标准 AI 审查口径。

## 14. 复现目标

一个外部执行主体如果正确读取并执行本知识系统，应能复现 Codex 审查的主要方法：

- 不是只检索关键词；
- 不是只输出大类风险；
- 会先画像和路由；
- 会逐章读取；
- 会按动作协议执行；
- 会拆分原子风险；
- 会用质量门反查漏项；
- 会记录未命中动作；
- 会保留原文证据；
- 会把审查结论反链到知识。

这就是本知识系统对外提供的核心能力。

# 政府采购招标文件业务审查流水线

类型:: 审查协议
状态:: 有效
主题:: 业务审查主流程; 审查流水线; 政府采购招标文件
适用范围:: 政府采购货物和服务招标文件合规审查
所属闭环:: 业务审查主流程线
关联总纲:: [[wiki/00-入口/四线闭环知识系统设计]]; [[wiki/00-入口/动作型知识系统建设纲领]]
关联执行规范:: [[wiki/70-审查协议/知识驱动审查执行规范]]
关联复现规范:: [[wiki/70-审查协议/Codex审查过程复现规范]]
关联矩阵:: [[wiki/20-知识点/政府采购逐章审查矩阵]]

## 1. 定位

本流水线是四线闭环中的第二条线：业务审查主流程线。

它回答“收到一份新的政府采购招标文件后，如何利用本 LLM Wiki 的知识完成审查”。这里描述的是 Codex 实际执行审查时采用的方法、流程、判断规则和产物要求。外部执行主体可以读取本流水线，按同样逻辑模拟 Codex 的审查过程。

本流水线不绑定任何具体执行器、命令或系统实现。它只定义业务审查动作。

## 2. 总体流程

```text
新文件进入
  -> 原文抽取
  -> 文件画像
  -> 知识路由
  -> 章节拆解
  -> 动作任务生成
  -> 逐章逐动作审查
  -> 风险原子化
  -> 证据链补齐
  -> 质量门反查
  -> 审查记录生成
  -> 运行记录生成
  -> 人工复核入口
  -> 复盘和知识更新入口
```

每一步都必须有输入、调用知识、执行动作和产物。

## 3. 输入边界

### 3.1 允许使用

- `raw/` 中的原始招标文件；
- `raw/抽取文本/` 中已抽取的文本；
- `wiki/` 中已有的法规、行业基础、知识点、风险审查点、风险库、审查协议、模板和语料；
- `tools/`、`ingest`、`query`、`lint` 等本地维护工具；
- 用户明确允许使用的材料。

### 3.2 禁止使用

标准 AI 审查不得使用：

- 外部标准答案；
- V2 检查结果；
- 人工批注文件；
- 未入库的历史标注；
- 其他目录中的同项目检查结果；
- 审查前从外部样本反推风险。

外部材料只能用于审后精确率提升，不得混入业务审查主流程。

## 4. 阶段一：原文抽取

### 输入

- 原始招标文件；
- 已存在的抽取文本或来源笔记。

### 调用知识

- [[wiki/60-提示词/条款抽取提示词]]
- [[wiki/15-行业基础/招标文件结构]]

### 执行动作

- 将 Word、PDF 或其他格式转为可检索文本；
- 保留章节标题、表格结构、页码或行号；
- 标记公告、需求、评分、合同、格式等主要章节；
- 对金额、日期、编号、资格条件、评分项和合同空白保持原样。

### 产物

- 抽取文本；
- 来源笔记；
- 原文定位能力。

### 容易出错

- 表格列错位；
- 金额合计错误；
- 行号缺失；
- 合同附件未抽取；
- 空白项被忽略。

## 5. 阶段二：文件画像

### 输入

- 原文抽取文本；
- 招标公告和项目基本信息；
- 采购计划、服务需求明细和政策条款。

### 调用知识

- [[wiki/20-知识点/政府采购招标文件画像]]
- [[wiki/20-知识点/采购品目和标的属性审查]]
- [[wiki/15-行业基础/政府采购专项场景画像]]

### 执行动作

抽取并判断：

- 项目编号、项目名称、采购人、代理机构；
- 采购方式、评审方法、是否评定分离；
- 采购品目、采购标的、数量、单位、所属行业；
- 预算金额、最高限价、分项预算和采购计划编号；
- 是否专门面向中小企业、是否接受联合体、是否允许分包、是否允许进口产品；
- 场景标签，例如学校物业、医院物业、法院机关物业、信息化设备；
- 文件章节完整性；
- 初步风险入口。

### 产物

- 文件画像表；
- 知识路由标签；
- 初步审查重点。

### 质量要求

画像不能只是摘要，必须能解释后续为什么调用某些知识和动作协议。

## 6. 阶段三：知识路由

### 输入

- 文件画像；
- 项目地域、采购方式、品类、场景和章节信息。

### 调用知识

- [[wiki/20-知识点/知识分层与路由规则]]
- [[wiki/10-法规依据/法规采集清单]]
- [[wiki/25-风险审查点/风险审查点总览]]
- 命中的品类动作包和审查协议。

### 执行动作

按顺序调用：

```text
国家通用规则
  -> 地方规则
  -> 采购方式规则
  -> 标的属性规则
  -> 品类规则
  -> 场景规则
  -> 章节规则
  -> 试运行规则
```

物业管理项目至少调用：

- [[wiki/15-行业基础/物业管理服务采购背景]]
- [[wiki/20-知识点/物业管理动作化审查包]]
- [[wiki/70-审查协议/物业管理审查动作协议]]
- 人员、认证、资质、业绩、荣誉、软著、合同、响应时限等风险审查点。

### 产物

- 调用知识清单；
- 本次审查适用规则说明；
- 不适用知识说明。

## 7. 阶段四：章节拆解

### 输入

- 原文抽取文本；
- 文件目录；
- 逐章审查矩阵。

### 调用知识

- [[wiki/20-知识点/政府采购逐章审查矩阵]]
- [[wiki/20-知识点/审查作业清单]]

### 执行动作

将文件拆为：

- 招标公告；
- 投标人须知；
- 资格条件；
- 资格性和符合性审查；
- 采购需求或用户需求书；
- 评分办法；
- 政府采购政策；
- 投标文件格式；
- 合同条款；
- 付款、验收和考核附件。

每章形成“已读、已抽取、已判断”的状态。

### 产物

- 逐章审查记录；
- 关键条款摘录；
- 缺失项列表。

## 8. 阶段五：动作任务生成

### 输入

- 文件画像；
- 章节拆解结果；
- 品类动作协议；
- 逐章审查矩阵。

### 调用知识

- [[wiki/70-审查协议/政府采购招标文件审查协议]]
- [[wiki/70-审查协议/物业管理审查动作协议]]
- [[wiki/70-审查协议/知识驱动审查执行规范]]

### 执行动作

生成本次动作清单。每个动作至少包括：

| 字段 | 内容 |
| --- | --- |
| 动作ID | 例如 `PROP-A09` |
| 动作名称 | 与动作协议一致 |
| 必读章节 | 本动作需要读取的原文范围 |
| 触发信号 | 原文中需要查找的信号 |
| 必须检查 | 成立条件、排除条件、缺失判断 |
| 执行状态 | 命中、未命中、待确认 |
| 关联审查点 | 稳定风险审查点 |
| 输出要求 | 风险标题、原文、问题说明和建议口径 |

### 产物

- 动作清单；
- 未命中也
